# Methodology

## Research question

What is the cost/time tradeoff of dispatching isolated coding tasks to different Claude
model tiers (Haiku, Sonnet, Opus) when used as subagents under project-manager
orchestration?

## Experimental setup

- **8 tasks** defined as frozen JSON in `tasks/001-*.json` through `tasks/008-*.json`.
  Each task is a self-contained coding task in a PHP/Slim mock project
  (`mock-project/`).
- **3 model tiers:** Claude Haiku, Sonnet, and Opus. Exact model IDs are pinned
  at experiment start via `runner/bin/cli state pin-models` and stored in
  `state.json`. This guards against silent model upgrades between dispatches.
- **N=3 replicates** per (task, tier) cell. Total subagent dispatches: 8 × 3 × 3 = 72.
- **Dispatch order** is shuffled per task using a seeded Fisher–Yates
  (`mt_srand(plan_seed ^ crc32(task_id))`, plan_seed=42). Order is deterministic
  and reproducible.
- **Policy A (retry-only):** on failed evaluation, the same-tier subagent retries
  up to 3 iterations within the same run. This is what is actually executed.
- **Policy B (cheapest-first escalation)** is *simulated* from Policy A data,
  not run directly. See `findings.md` and the "Policy B simulation" section below.

## Metrics

Recorded in `results/results.jsonl` (one JSONL row per run):

- **Primary:** `wall_clock_subagent_s` — wall-clock seconds the subagent was
  working, excluding PM-side orchestration.
- **Secondary:** `tokens_subagent_in + tokens_subagent_out` — total subagent
  tokens (input + output), excluding PM overhead.
- **Outcome:** `passed` or `failed`, as determined by the evaluator's check
  suite for the task.

PM-side overhead (`tokens_pm_overhead`) is recorded separately and reported but
not used in tier comparisons, because it is structurally per-task and not
per-tier.

## Evaluator checks

Seven check types, implemented in `runner/src/Evaluator/Check/`:

1. **`phpunit`** — run a subset of the mock-project's PHPUnit suite and assert
   all tests pass.
2. **`query_count`** — hit a route via the query-count probe; assert total
   MariaDB queries are at or below a task-specified limit.
3. **`smoke_no_regressions`** — run the full mock-project smoke suite; assert
   no tests that were previously passing are now failing.
4. **`lint`** — PHPStan level 6 on mock-project code.
5. **`file_exists`** — assert a list of files/paths exist.
6. **`grep_not_present`** — assert a regex does not match in a set of paths.
7. **`diff_size_limit`** — assert the diff from a base ref is under N changed
   lines.

A run's `outcome` is `passed` iff all configured checks pass.

## Policy B simulation (bootstrap)

Policy B is *cheapest-first tier escalation*:

> Start at haiku. If the evaluator says "failed," retry at sonnet. If that
> fails, retry at opus. If that fails too, record "max_tier_failed" for the task.

We did not run Policy B directly — that would cost another ~72 dispatches.
Instead we simulate it from the observed Policy A data using a **seeded
bootstrap Monte Carlo** (see `runner/src/Analysis/BootstrapSimulator.php`):

```
for 1000 iterations (seeded with plan_seed=42):
  for each of the 8 tasks:
    for tier in [haiku, sonnet, opus]:
      sample one of the 3 recorded runs for (task, tier) with replacement
      add that run's tokens and wall-clock time to the task total
      if outcome == passed: continue to next task
    if no tier passed: count this task as max_tier_failed
```

Over 1000 iterations, we get a distribution of totals (tokens and wall-clock
seconds). We report the mean and the 2.5th / 97.5th percentiles as a 95%
confidence interval.

**Assumption:** the 3 observed runs per cell are representative of the
underlying distribution. With N=3 this assumption is fragile. See `limitations.md`.

## Reproducibility

The entire pipeline is deterministic end-to-end:

- `plan_seed=42` drives both the run-order shuffle and the bootstrap sampling.
- PHP dependencies are pinned via `composer.lock`.
- The mock project is a frozen git commit; the evaluator reads a frozen task bank.
- MariaDB runs in Docker (10.11) with a fixed init script.
- `findings.md` is regenerated by `runner/bin/cli report` and committed; any
  reviewer can diff their regenerated copy against the committed file.

Anything non-reproducible (e.g. server-side model tuning) is documented in
`limitations.md`.
